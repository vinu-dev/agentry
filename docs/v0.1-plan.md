# Agentry — v0.1 Build Plan

Status: **draft, pre-implementation**

The lean version. v0.1 is a weekend project, not a 10-week framework.

Background: [`docs/architecture.md`](architecture.md). Practical guide: [`docs/how-to-use.md`](how-to-use.md).

---

## 1. Goal

> An Operator on Linux or Windows installs Agentry as a CLI tool. They go to a target repo (e.g., rpi-home-monitor), run `agentry init` to add `.agentry/config.yml` + `docs/ai/roles/*.md` skeletons, edit the config to pick which model handles each role, edit the rule files to define what each role does, and run `agentry start`. The orchestrator detects which CLIs are installed (claude, codex, etc.), spawns one forever-loop per declared role, and runs them in parallel forever. Discord pings on every event. The system survives reboots via systemd / NSSM. Any number of roles supported (6 for hobby projects, 11+ for medical device).

If we ship that, v0.1 is done.

## 2. Module structure

```
src/agentry/
├── __init__.py
├── version.py              # __version__
├── cli.py                  # `agentry start | stop | status | doctor | init | target add`
├── config.py               # load + validate .agentry/config.yml + pipeline.local.toml
├── prompt.py               # GENERIC_PROMPT template + role substitution
├── orchestrator.py         # role_loop function + thread spawner
├── supervisor.py           # subprocess + dual timeout (stall + total)
├── notify.py               # Discord webhook poster, batched
├── platform/
│   ├── linux.py            # systemd unit generator
│   └── windows.py          # NSSM XML generator
└── github.py               # tiny gh CLI wrapper for label/PR queries
```

That's the whole framework. ~7 modules, ~250 lines total.

## 3. What ships in v0.1

| Capability | How |
|------------|-----|
| **N forever-loops in parallel** (any number) | One `threading.Thread` per declared role |
| **Per-role model assignment** | `cli` field in `.agentry/config.yml` |
| **Per-role timeouts** | `interval_min`, `total_min`, `stall_min` |
| **Generic prompt** (framework-provided) | `prompt.py` builds it from template + role list |
| **Process supervision** | `Popen` + dual timeout in `supervisor.py` |
| **CLI auto-detection** | `agentry doctor` checks every declared `cli` is on PATH |
| **Non-interactive flags** | `args:` field in config |
| **Discord notifications** | webhook POST, batched 60s |
| **systemd / NSSM service** | `agentry service install` writes the unit |
| **Restart-safe** | no in-memory state, GitHub is the source of truth |
| **`agentry doctor`** | checks config schema, role files, CLIs on PATH, labels in target repo |
| **`agentry start / stop / status`** | tiny CLI |
| **Hardware support** | implicit — agents have shell/ssh/curl access; hardware steps go in `docs/ai/roles/tester.md` |

## 4. What does NOT ship in v0.1

Explicitly deferred:

| Feature | When | Why |
|---------|------|-----|
| Babysitter LLM (read agent stdout, decide actions) | v0.2 | Add only after seeing real failure patterns |
| Telegram / email notifiers | v0.2 | Discord first |
| Auto-promote rules (e.g., issue auto-labeled `ready-for-design` after 24h with no veto) | v1+ | Operator decides what's worth working on |
| Multi-target parallelism (concurrent dispatch across multiple targets) | v1+ | v0.1 supports one target at a time |
| Hardware-specific helpers (smart-plug recovery, SD flasher) | not planned | Agents call them via shell from role rule files; no framework code needed |
| Dedicated Yocto / SWU / FDA-submission tooling | not planned | Lives in role rule files (`release.md`, `regulatory_reviewer.md`); framework just dispatches |

## 5. Acceptance criteria

v0.1 is **done** when these scenarios pass on Ubuntu 24.04 and Windows 11:

### 5.1 Cold install

```bash
uv tool install --from git+ssh://git@github.com/vinu-dev/agentry.git agentry
agentry --version
mkdir -p ~/.agentry/state
$EDITOR ~/.agentry/.env                                    # API keys + GitHub PAT + Discord webhook
$EDITOR ~/.agentry/pipeline.local.toml                     # state dir + token env names
agentry service install                                    # systemd or NSSM
agentry status                                             # services running
```

### 5.2 Onboard a target (gtest-style integration)

```bash
cd ~/some-target-repo                                     # any repo
agentry init                                               # creates .agentry/config.yml + docs/ai/roles/*.md skeletons (6 default roles)
$EDITOR .agentry/config.yml                                # pick models per role
$EDITOR docs/ai/roles/{researcher,architect,implementer,tester,reviewer,release}.md
git commit -am "Add Agentry config and role rules"
git push
agentry doctor --target git@github.com:user/some-target-repo.git
# Validates: config valid, all role files present, all CLIs on PATH, all labels exist
agentry target add --repo git@github.com:user/some-target-repo.git
agentry status                                             # shows N role threads spinning
```

### 5.3 N-role roster (medical device example)

```bash
cd ~/medical-device-repo
agentry init --template medical-device                     # uses 11-role template from docs/examples/medical-device/
# now have: researcher, risk_analyst, architect, implementer, tester,
#           code_reviewer, quality_reviewer, cybersecurity_reviewer,
#           regulatory_reviewer, traceability_tracker, release
$EDITOR .agentry/config.yml                                # confirm models
agentry doctor --target git@github.com:user/medical-device-repo.git
agentry target add --repo ...
agentry status                                             # 11 role threads running
```

The framework code is identical between the 6-role and 11-role cases.

### 5.4 One full task round-trip (with the standard 6 roles)

```
1. Operator opens an issue manually, labels `ready-for-design`
2. Within 5 min: Architect writes design doc, opens PR (auto-merged docs-only),
   relabels `ready-for-implementation`
3. Within 5 min: Implementer codes on a branch, relabels `ready-for-test`
4. Within 5 min: Tester runs `pytest`, green → opens PR `ready-for-review`
5. Within 5 min: Reviewer reads diff, approves
6. GitHub auto-merge fires
7. Discord shows the full sequence
```

### 5.5 Stall recovery

```
1. Force a CLI to hang (e.g., point it at a non-responsive endpoint)
2. After `stall_min` minutes, orchestrator kills it
3. Discord: "<role>: stalled (silent Xm)"
4. Next interval, fresh subprocess spawns
```

### 5.6 Total-timeout recovery

```
1. Force a CLI to take longer than `total_min`
2. After `total_min`, orchestrator kills it
3. Discord: "<role>: total timeout (Xm)"
4. Next interval works
```

### 5.7 Service restart

```
1. Kill the daemon process
2. systemd / NSSM restarts it within 10s
3. All N role threads resume; GitHub state intact
```

### 5.8 Cross-platform parity

All of the above pass on Ubuntu 24.04 (systemd) AND Windows 11 (NSSM). Same Python, same config files.

### 5.9 Hardware integration via rule file

```
1. tester.md says: "SSH to 192.168.1.50, run `systemctl status app`,
   capture output, label result"
2. Tester agent reads tester.md, executes ssh, parses output
3. Agentry doesn't need any hardware-specific code; the agent uses shell tools
```

## 7. Test strategy

Unit tests for: config validation (any-number-of-roles), supervisor timeouts, prompt template substitution, Discord batching, label name resolution, CLI auto-detection.

One integration test: a synthetic target repo with stub role files that just `echo hello && exit 0`. Confirm all declared roles spawn, exit cleanly, and Discord receives the events. Run with N=3 roles, N=6 roles, and N=11 roles to confirm the framework scales.

That's it. No mock LLM responses, no recorded fixtures. The framework doesn't care what the agents do — only that they start, run within timeout, and exit.

## 8. Risks

1. **CLI version drift.** `claude` and `codex` flags change. Mitigation: pin tested CLI versions in docs; operator chooses which version to install.
2. **Discord rate limits.** 30 msg/min/webhook. Mitigation: 60s batch flush.
3. **Subprocess kill on Windows.** Process tree termination is awkward. Mitigation: `taskkill /T /F`. ~10 lines of platform-specific code.
4. **OAuth credentials for `claude` / `codex` subscriptions.** Service must run as Operator's user, not SYSTEM. Mitigation: NSSM `ObjectName` config in installer; documented in `how-to-use.md`.
5. **Role file ambiguity.** Repo owners may write unclear rule files; agents misbehave. Mitigation: not the framework's problem — the framework just dispatches. `agentry doctor` flags missing files but not unclear ones.
6. **Many roles fighting for context.** With 11+ roles each running an LLM call every 5 min, costs and rate limits matter. Mitigation: tune `interval_min` higher for heavy projects; use subscription routing.

No risks involve our code being complex, because there isn't much.
