# Skynet Agentry — v0.1 Build Plan

Status: **draft, pre-implementation**

This document is the concrete build plan for v0.1 — the first version of Skynet Agentry that is operator-usable end-to-end. It assumes you have read [`docs/architecture.md`](architecture.md) and accepted the architectural decisions there.

v0.1 follows the v0.0a spec series (architecture, compatibility spec, schema, pipeline template, env template), all committed already.

---

## 1. v0.1 Goal Statement

> An Operator on a Linux or Windows host can install Skynet Agentry, onboard a target repository, and watch the system ship a feature end-to-end without manual intervention — Researcher creates an issue, Architect writes a design, Implementer codes it, Tester runs the configured tests, PR Author opens a PR, Reviewer (different vendor) approves, auto-merge fires according to path policy, Discord notifies. The system survives agent crashes, rate limits, capability gaps, and stalls without operator intervention.

If we ship this, Skynet Agentry exists.

---

## 2. In Scope (v0.1)

**Runtime:**

- `skynet-orchestrator` daemon (Python + APScheduler + sqlite)
- `skynet-watchdog` daemon (deterministic supervisor)
- Service installers: systemd unit files (Linux) + NSSM XML (Windows)
- sqlite schema with task_counters, tasks, events, heartbeats, budget
- Crash-safe checkpoint resume

**Provider abstraction:**

- Built-in: `anthropic`, `openai`, `codex_cli`
- Custom: `openai_compatible`, `anthropic_compatible`, `claude_cli`, `codex_cli`-as-custom
- Fallback chain mechanics (primary → fallback1 → fallback2 → ...)
- Multi-vendor enforcement (implementer ≠ reviewer vendor)
- Subscription routing via `claude` and `codex` CLIs

**Failure handling:**

- The four modes (CRASH, CAPABILITY_EXCEEDED, RATE_LIMITED, STALLED), all deterministic
- Per-role overrides via `.skynet/config.yml`
- Budget enforcement as 4th boundary (daily $, per-task tokens, retry cap, concurrency cap, research-issues-per-day cap)

**Agent roster (all 7):**

- Researcher (cron, drafts issues)
- Architect (design docs)
- Implementer (code + unit tests)
- Tester (runs configured commands)
- PR Author (opens PRs with structured front-matter)
- PR Reviewer (different vendor, approves/requests changes)
- Release Engineer (defined but stubbed — emits "v1+ feature" if invoked)

Default prompts in `prompts/<role>.md`, distributed with the framework via `skynet bootstrap`.

**Target onboarding:**

- `skynet init --target <repo>` opens compliance-scaffold PR
- `skynet doctor --target <repo>` validates against COMPATIBILITY-SPEC.md
- Version handshake (`skynet_version` range)

**Task lifecycle:**

- Full state machine through `merged` and `quarantined`
- Path-policy auto-merge with most-restrictive resolution
- docs-only / tests-only allowlist guard rule
- Soft-hold timer with Operator veto via `skynet pause --task`

**Trust boundaries:**

- Path allowlist (workspace cwd only)
- Command allowlist (per-role)
- Network allowlist (configured providers + github.com + Researcher web sources)
- Budget enforcement (host-wide + per-target)
- Audit log + reject-burst quarantine

**Notifications:**

- Discord webhook, batched 60s, immediate flush for critical events

**Operator CLI:**

- `skynet --version`
- `skynet bootstrap` (refresh prompts/policies)
- `skynet service install` (systemd or NSSM)
- `skynet init --target <repo>`
- `skynet doctor [--target <repo>]`
- `skynet target add / list / remove`
- `skynet status [--target <repo>] [--task <id>]`
- `skynet logs <task-id> [--follow]`
- `skynet pause / resume [--target | --task]`
- `skynet kick <agent>`
- `skynet replay <task-id> [--from <state>]`
- `skynet quarantine <task-id> [--close]`
- `skynet config get / set / show / providers list`
- `skynet promote <issue-id>` (research-draft → designed)
- `skynet unlock <policy>` (requires auth token)
- `skynet notify test` (Discord ping verification)
- `skynet recover` (rebuild state from GitHub if DB is lost)
- `skynet migrate --target <repo> --from <ver> --to <ver>` (stub for v0.1; full impl v1+)

## 3. Out of Scope (v0.1)

Explicitly **not** delivered in v0.1:

- Hardware integration — Tester runs configured commands only, no flash, no SSH to a Pi, no serial scrape
- Yocto release engineering — Release Engineer agent is defined but stubs out
- SWU bundle signing — reserved for v1+
- Smart-plug power-cycle recovery — reserved for v1+
- GCP VM offload — `[gcp_vm].enabled` must remain false
- Multi-target *parallelism* (concurrent tasks across targets) — v0.1 supports multiple targets but serializes work
- Telegram or email notifiers — Discord only
- Public PyPI publication — private repo, `git+ssh` install only
- A CLI on iOS/Android — desktop only
- A web dashboard — CLI only
- Auto-promote rules (research-draft → designed without operator input) — v1+ feature; v0.1 always requires explicit promotion

These constraints are enforced at multiple layers: schema validation, runtime checks, and explicit error messages.

## 4. Acceptance Criteria

v0.1 is **done** when these end-to-end scenarios pass on a fresh Linux host (Ubuntu 24.04) and a fresh Windows host (Windows 11):

### 4.1 Cold install

```
[fresh host, no .skynet/, no services]
1. uv tool install skynet-agentry
2. skynet --version                                          → prints version
3. skynet bootstrap                                          → populates ~/.skynet/{prompts,policies}
4. echo secrets >> ~/.skynet/.env
5. cp pipeline.example.toml ~/.skynet/pipeline.local.toml; edit
6. skynet service install                                    → services running
7. skynet status                                             → orchestrator/watchdog running, 0 targets
```

### 4.2 Target onboarding

```
[from above, on a clean target repo]
1. skynet init --target <repo>                              → compliance-scaffold PR opened
2. [operator merges PR, edits config.yml]
3. skynet doctor --target <repo>                            → PASS, level: standard
4. skynet target add --repo <repo>                          → orchestrator picks it up
5. [wait up to 1 hour for cron]
6. GitHub: new issue with skynet/research-draft label
```

### 4.3 Full task round-trip (docs-only feature, no real hardware)

```
[from above, with a docs-only research-draft issue]
1. skynet promote <issue-num>                                → label flips to skynet/designed
2. [wait]
3. New PR: design doc commit, auto-merged (docs-only = hard policy)
4. [wait]
5. New PR: code+test branch, auto-merged after Reviewer approval (or held 24h)
6. Discord pings throughout
7. skynet status --task <task-id>                            → state: merged
```

### 4.4 Failure recovery

Scripted scenarios that exercise each failure mode:

- **CRASH:** kill an agent worker mid-task; orchestrator restarts from checkpoint, task completes
- **CAPABILITY_EXCEEDED:** force a model to fail 3× same diff; orchestrator escalates to fallback model, task completes
- **RATE_LIMITED:** stub a provider to return 429; orchestrator backoffs, falls back, task completes
- **STALLED:** pause an agent's heartbeat updates; orchestrator kills + restarts once, then quarantines on second stall

Each scenario is a test in `tests/integration/recovery/`.

### 4.5 Multi-vendor enforcement

```
1. Edit .skynet/config.yml: set both implementer and reviewer to anthropic
2. skynet doctor --target <repo>                             → FAIL with MULTI_VENDOR_VIOLATION
3. Orchestrator refuses to dispatch tasks to this target until fixed
```

### 4.6 Budget enforcement

```
1. Set budget.daily_dollars to a low value (e.g. $5)
2. Run tasks until budget exhausted
3. Discord: budget.exhausted notification (immediate, not batched)
4. Orchestrator pauses all dispatch
5. skynet unlock budget --token <op-token>                   → resumes
```

### 4.7 Operator override paths

```
- skynet pause --task <id>      → soft-hold PR is vetoed, no auto-merge
- skynet replay <id> --from agent-ready   → quarantined task restarts cleanly
- skynet kick implementer       → running implementer is killed and respawned
```

### 4.8 Cross-platform

All of the above pass on:

- Ubuntu 24.04 LTS (systemd)
- Windows 11 Pro (NSSM + WSL2)

If any criterion fails, v0.1 is not done.

## 6. Build Order

Strict dependency order — each phase depends on the previous. No phase begins until its predecessor is committed and tested.

### Phase 1 — State + Config Foundation (week 1)

Smallest layer that everything else depends on.

```
state/schema.py                     CREATE TABLE definitions
state/migrations.py                  versioned schema, upgrade path
state/repo.py                        data access (tasks, events, heartbeats, budget)
state/task_id.py                     atomic SKY-YYYY-NNNNNN allocator
config/schema.py                     pydantic models from JSON Schema
config/loader.py                     YAML + TOML → merged config object
config/resolver.py                   role + fallback resolution
```

Tests: round-trip a config through loader and resolver; allocate 100 task IDs across two simulated years; concurrent ID allocation safety; checkpoint serialize/deserialize.

### Phase 2 — Provider Layer (week 1–2)

Pluggable model access, fallback chain.

```
providers/base.py                    ProviderInterface ABC
providers/anthropic.py               Anthropic API client wrapper
providers/openai.py                  OpenAI API client wrapper
providers/openai_compatible.py       generic OpenAI-shape
providers/anthropic_compatible.py    generic Anthropic-shape
providers/claude_cli.py              subprocess wrapper for `claude -p`
providers/codex_cli_builtin.py       subprocess wrapper for `codex` (built-in)
providers/codex_cli_custom.py        custom-instance wrapper
providers/registry.py                name resolution, fallback chain
```

Tests: each provider exchanges a known prompt for a known response (with mocked HTTP); fallback chain falls through correctly on simulated failures; multi-vendor rule rejects same-vendor implementer/reviewer.

### Phase 3 — Failure Handling (week 2)

The four modes, classifier, recovery actions.

```
failure/modes.py                     enum
failure/exit_codes.py                signaling protocol
failure/classifier.py                signals → FailureMode (deterministic)
failure/recovery.py                  FailureMode → action
```

Tests: every failure mode classifies correctly from each kind of signal (exit code, heartbeat age, HTTP status, repeated diff); recovery actions invoke the right next step (restart from checkpoint vs. escalate vs. fallback).

### Phase 4 — Agent Runner + Tool Wrapper (week 2–3)

The bridge between orchestrator and a worker subprocess.

```
agents/tool_wrapper.py               path/command/network/budget allowlists
agents/runner.py                     AgentRunner — spawns subprocess, manages lifecycle
agents/prompts.py                    prompt loading
agents/checkpoints.py                checkpoint persistence
```

Tests: agents can read/write inside their workspace; rejected tool calls produce structured errors; checkpoint round-trip preserves agent state; tool wrapper integrates with budget tracker.

### Phase 5 — Orchestrator Daemon (week 3–4)

The long-running loop.

```
orchestrator/scheduler.py            APScheduler integration
orchestrator/policy_engine.py        path-policy resolution
orchestrator/label_watcher.py        GitHub label polling
orchestrator/dispatcher.py           task → agent → subprocess
orchestrator/ipc.py                  CLI ↔ daemon socket
orchestrator/daemon.py               __main__
```

Tests: orchestrator boots and halts cleanly; CLI commands reach the daemon over IPC; one task moves through the full lifecycle with mocked agents.

### Phase 6 — Watchdog Daemon (week 4)

Two-tier supervision.

```
watchdog/heartbeat.py                heartbeat polling
watchdog/recovery_actions.py         restart/kill/escalate
watchdog/budget.py                   daily $ tracker
watchdog/daemon.py                   __main__
```

Tests: watchdog detects orchestrator-down and exits with retryable code (init system restarts it); detects agent-down and triggers recovery; budget exhausted pauses dispatch.

### Phase 7 — Notifier (week 4–5)

Discord, batched.

```
notifier/events.py                   event taxonomy
notifier/batcher.py                  60s flush
notifier/discord.py                  webhook POST

```

Tests: events accumulate and flush every 60s; critical events bypass the batch; rate-limit handling for Discord's per-webhook 30/min cap.

## 7. Test Strategy

| Layer | Approach | Coverage target |
|-------|----------|-----------------|
| Unit | pytest, mock external dependencies | 90%+ on `config/`, `providers/`, `failure/`, `state/` |
| Integration | pytest with synthetic target repos and recorded provider responses | every acceptance scenario in §4 |
| End-to-end | shell scripts that exercise the CLI on real Ubuntu/Windows VMs | the full §4 acceptance flow |
| Failure injection | dedicated tests for each failure mode | each FailureMode has a deterministic recovery test |
| Cross-platform | CI matrix: Ubuntu 24.04, Windows 11 | every test runs on both |

The mocked provider responses live in `tests/fixtures/responses/`. For each agent, we have canned responses representing:

- happy path
- capability-exceeded signal
- a model output that triggers each failure mode
- malformed responses (provider misbehaves)

This makes the tests deterministic and fast; real LLMs are only invoked in optional smoke tests gated by env var.

## 8. Key Risks

### 8.1 Subscription routing fragility

`claude` and `codex` CLIs are designed for interactive use. Running them headlessly in subprocesses might break with future versions. Mitigation:

- Pin to a known-working CLI version range
- Provide explicit fallback to API-key providers in default configs
- Document which CLI versions have been tested

### 8.2 Cross-platform timing

NSSM signal handling differs from systemd. A graceful-stop sequence that works perfectly on Linux may leave processes around on Windows. Mitigation:

- Each platform has its own kill-tree implementation in `platform/subprocess.py`
- Integration tests run on both OSes from week 1; no "Linux-first development, Windows-port-later" — that pattern always produces 6 months of Windows bugs

### 8.3 sqlite under heavy concurrency

Several agents writing heartbeats simultaneously, plus the orchestrator writing tasks, plus the watchdog reading. WAL mode helps but isn't infinite. Mitigation:

- All writes go through the orchestrator (heartbeats are written to per-agent files first, then orchestrator promotes to sqlite)
- Watchdog is read-only on the DB (uses heartbeat files for liveness)
- Concurrent task cap (default 4) keeps it manageable

### 8.4 Prompt injection via researched content

Researcher pulls web content. A malicious page could try to escape into the agent's tool calls. Mitigation:

- Content is summarized into structured fields (title, key points, sources) before being passed to other agents
- Tool wrapper rejects any path / command / network call that wasn't in the original allowlist, no matter what the LLM "asks for"
- audit-reject-burst quarantines the task before damage compounds

### 8.5 Budget overrun before watchdog notices

Race condition: an agent makes 50 fast LLM calls in 5 seconds, blowing through the daily budget before the watchdog's 60s poll fires. Mitigation:

- Budget tracker runs in-process inside the orchestrator (not as a separate process)
- Each tool call increments tokens-used synchronously; cap check happens in the same path
- Watchdog's 60s poll is the safety net, not the primary enforcement

### 8.6 GitHub API rate limits

7 agents making API calls (labels, PRs, issues) can hit GitHub's per-token rate limit. Mitigation:

- All `gh` calls go through a shared rate-limiter
- Conditional requests with ETags where possible
- Backoff on 403/429

### 8.7 Operator-token leakage

`skynet unlock` requires a token. If an agent ever obtains it, the budget cap and policy enforcement collapse. Mitigation:

- Token lives in a file at `~/.skynet/.unlock-token` with mode 0600
- Tool wrapper has `~/.skynet/` in its `forbidden_paths` list permanently
- Unlock command only accepts the token via stdin (not args, not env), reducing leak surfaces

## 10. Success Definition

**v0.1 is successful if** an Operator can:

1. Install Skynet Agentry on a fresh host in under 30 minutes
2. Onboard a target repository in under 30 minutes
3. Walk away for a week and return to find features merged on the target without intervention
4. Trust that no merge violated the configured path policies
5. Trust that no agent ran outside its workspace
6. Trust that costs stayed within the configured daily cap
7. Recover from any failure mode by reading Discord pings and (rarely) running a single CLI command

If all seven hold for two consecutive weeks operating on `rpi-home-monitor`, v0.1 is shipped.
