# Skynet Agentry — v0.1 Build Plan

Status: **draft, pre-implementation**

The lean version. v0.1 is a weekend project, not a 10-week framework.

Background: [`docs/architecture.md`](architecture.md). Practical guide: [`docs/how-to-use.md`](how-to-use.md).

---

## 1. Goal

> An Operator on Linux or Windows installs Skynet Agentry as a CLI tool. They configure `.skynet/config.yml` in a target repo (rpi-home-monitor) and write `docs/ai/roles/*.md` for each of the 6 roles. They run `skynet start`. The 6 roles run in parallel forever, each spawning its assigned LLM CLI on its interval, until the operator runs `skynet stop`. Discord pings on every event. The system survives reboots via systemd / NSSM.

If we ship that, v0.1 is done.

## 2. Module structure

```
src/skynet_agentry/
├── __init__.py
├── version.py              # __version__
├── cli.py                  # `skynet start | stop | status | doctor | init`
├── config.py               # load + validate .skynet/config.yml + pipeline.local.toml
├── orchestrator.py         # the role_loop function + thread spawner
├── supervisor.py           # subprocess + timeout helpers
├── notify.py               # Discord webhook poster, batched
├── platform/
│   ├── linux.py            # systemd unit generator
│   └── windows.py          # NSSM XML generator
└── github.py               # tiny gh CLI wrapper for label/PR queries
```

That's the whole framework. ~6 modules, ~200-300 lines total.

## 3. What ships in v0.1

| Capability | How |
|------------|-----|
| 6 forever-loops in parallel | `threading.Thread` per role, daemon=True |
| Per-role model assignment | `cli` field in `.skynet/config.yml` |
| Per-role timeouts | `interval_min`, `total_min`, `stall_min` |
| Process supervision | `Popen.wait(timeout=)`, `proc.kill()` on stall |
| Non-interactive flags | `args:` field in config |
| Discord notifications | webhook POST, batched 60s |
| systemd / NSSM service | `skynet service install` writes the unit |
| State on restart | none — GitHub is the state |
| `skynet doctor` | checks config schema, role files, labels, binaries |
| `skynet start / stop / status` | tiny CLI |

---

## 4. What does NOT ship in v0.1

Explicitly deferred:

| Feature | When | Why |
|---------|------|-----|
| Babysitter LLM (read agent stdout, decide actions) | v0.2 | Add only after seeing real failure patterns |
| Hardware test integration (flash Pi, serial scrape) | v1+ | Unrelated to the orchestrator core |
| Yocto build orchestration | v1+ | Repo's `release.md` rule file calls bitbake; framework just dispatches |
| Multi-target parallelism in one orchestrator | v1+ | v0.1 supports one target at a time |
| JSON Schema for config validation | maybe never | Pydantic in code is enough for a 30-line YAML |
| Provider abstraction with fallback chains in framework code | maybe never | Operator picks the binary; if it's down the role pauses |
| Telegram / email notifiers | v0.2 | Discord first |
| Auto-promote rules (research-draft → ready-for-design) | v1+ | Operator decides what's worth designing |

## 5. Acceptance criteria

v0.1 is **done** when these scenarios pass on Ubuntu 24.04 and Windows 11:

### 5.1 Cold install

```bash
uv tool install --from git+ssh://git@github.com/vinu-dev/skynet-agentry.git skynet-agentry
skynet --version
mkdir -p ~/.skynet/state
$EDITOR ~/.skynet/.env                                    # add API keys + GitHub PAT + Discord webhook
$EDITOR ~/.skynet/pipeline.local.toml                     # paths + token env var
skynet service install                                    # systemd or NSSM
skynet status                                             # services running
```

### 5.2 Target onboarding

```bash
cd ~/some-target-repo
$EDITOR .skynet/config.yml                                # 6 agents declared
mkdir -p docs/ai/roles
$EDITOR docs/ai/roles/{researcher,architect,implementer,tester,reviewer,release}.md
git commit -am "Add Skynet Agentry config and role rules"
git push
skynet doctor --target git@github.com:user/some-target-repo.git
# → PASS
skynet target add --repo git@github.com:user/some-target-repo.git
# Within 1 hour, Researcher fires; immediately, other roles start polling labels
```

### 5.3 One full task round-trip (with a docs-only feature)

```
1. Operator opens an issue manually, labels `ready-for-design`
2. Within 5 min, Architect picks up, writes design doc, opens design PR, relabels `ready-for-implementation`
3. Within 5 min, Implementer picks up, codes on a branch, relabels `ready-for-test`
4. Within 5 min, Tester runs configured tests (e.g., `pytest`), green, opens PR labeled `ready-for-review`
5. Within 5 min, Reviewer reads diff, approves
6. GitHub auto-merge fires (per repo's branch protection settings)
7. Discord shows the full sequence
```

### 5.4 Stall recovery

```
1. Force a CLI to hang (e.g., point it at a non-responsive endpoint)
2. After `stall_min` minutes of silence, orchestrator kills it
3. Discord ping: "<role>: stalled (silent Xm)"
4. Next interval, fresh subprocess spawns and succeeds
```

### 5.5 Total-timeout recovery

```
1. Force a CLI to take longer than `total_min`
2. After `total_min`, orchestrator kills it
3. Discord ping: "<role>: total timeout (Xm)"
4. Next interval works
```

### 5.6 Service restart

```
1. While orchestrator is running, kill the daemon process
2. systemd / NSSM restarts it within 10s
3. All 6 role threads resume; GitHub state intact; nothing lost
```

### 5.7 Cross-platform

All of the above pass on Ubuntu 24.04 (systemd) AND Windows 11 (NSSM). Same Python, same config files.

## 7. Test strategy

Unit tests for: config validation, supervisor timeouts, Discord batching, label name resolution.

One integration test: a synthetic target repo with stub role files that just `echo hello && exit 0`. Confirm all 6 roles spawn, exit cleanly, and Discord receives the events.

That's it. No mock LLM responses, no recorded fixtures. The framework doesn't care what the agents do — only that they start, run within timeout, and exit.

## 8. Risks

1. **CLI version drift.** `claude` and `codex` flags change. Mitigation: pin tested CLI versions in docs; operator chooses which version to install.
2. **Discord rate limits.** 30 msg/min/webhook. Mitigation: 60s batch flush.
3. **Subprocess kill on Windows.** Process tree termination is awkward. Mitigation: `taskkill /T /F` on Windows; SIGKILL + process group on Linux. ~10 lines of platform-specific code.
4. **OAuth credentials for `claude` / `codex` subscriptions.** Service must run as Operator's user account, not SYSTEM. Mitigation: NSSM `ObjectName` config in the installer; documented in `how-to-use.md`.
5. **Role file evolution.** Repo owners change their `docs/ai/roles/*.md` over time. That's fine; Skynet just reads them on every spawn.

No risks involve our code being complex, because there isn't much.
